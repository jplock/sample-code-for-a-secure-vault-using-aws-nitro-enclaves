This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

A secure vault for storing PII/PHI data using AWS Nitro Enclaves. Three-tier architecture:

1. **API Tier** (`api/`): Python Lambda functions using AWS Lambda Powertools, deployed via SAM. REST endpoints for vault CRUD; encrypts data with HPKE before storing in DynamoDB.
2. **Parent Tier** (`parent/`): Rust binary on EC2 running an Axum HTTP server. Manages enclave lifecycle via `nitro-cli`, caches IAM credentials from IMDSv2, proxies requests to the enclave over vsock.
3. **Enclave Tier** (`enclave/`): Rust binary inside a Nitro Enclave. Decrypts the symmetric key via KMS (over a vsock proxy), decrypts attributes with HPKE, and optionally applies CEL transformations.
4. **Canary** (`canary/`): Python Lambda that performs end-to-end health monitoring against the deployed API.

## Build & Test Commands

### Rust Workspace (root `Cargo.toml`)
```bash
cargo build                                          # Build both crates
cargo fmt --all -- --check                           # Format check (CI uses --check)
cargo clippy --all-features --all-targets -- -D warnings  # Lint (warnings denied in CI)
cargo test --verbose                                 # All tests
cargo test -p parent-vault                           # Tests for a single crate
cargo test -p enclave-vault test_name                # Single test by name
./scripts/check_no_panic.sh                          # Enforces no panic!/unreachable!/unimplemented! in enclave/src/
```

The enclave and parent crates set `unwrap_used`, `expect_used`, and `indexing_slicing` to `warn` in `[lints.clippy]`. Any necessary use of `expect` must be wrapped in `#[allow(clippy::expect_used)]` with a justification comment (see `parent/src/application.rs` for examples).

Property tests live alongside unit tests; regression cases are persisted in `enclave/proptest-regressions/` and `parent/proptest-regressions/` — commit any new regression files generated by `proptest`.

### Component-level Make targets
```bash
make -C enclave build         # cargo build for aarch64-unknown-linux-musl --release
make -C enclave build-docker  # Build enclave Docker image
make -C enclave build-enclave # nitro-cli build-enclave → enclave-vault.eif
make -C enclave run-enclave   # nitro-cli run-enclave (debug mode)
make -C parent build          # cargo build for aarch64-unknown-linux-gnu --release
make -C parent build-docker   # Build parent Docker image
```

### Python API / Canary
```bash
cd api && make setup    # python3 -m venv .venv + install dev/src/dependencies requirements
cd api && make build    # sam build --parallel --cached
cd api && make deploy   # sam deploy
cd api && make format   # black .
cd api && make outdated # List outdated packages
```
The same targets exist in `canary/`. Lambda layer dependencies live in `*/dependencies/requirements.txt`; runtime dependencies in `*/src/requirements.txt`.

### Docker multi-platform builds
```bash
docker buildx bake -f docker-bake.hcl  # Builds parent-vault:latest and enclave-vault:latest (linux/arm64)
# For reproducible builds, set SOURCE_DATE_EPOCH=$(git log -1 --format=%ct) before invoking
```

### Full stack deployment
```bash
./deploy.sh   # Orchestrates VPC → KMS → CI/CodePipeline → Vault → API stacks via CloudFormation
```
Requires a `.env` with `HOSTED_ZONE_ID`, `DOMAIN_NAME`, `GITHUB_REPO`. The IAM principal running this must have the `dp:exclude:network=true` tag.

## Architecture

```
Client → API Gateway → Lambda (Python)
                         ↓
                      DynamoDB (metadata/audit)
                         ↓
         NLB → NGINX → Parent (Rust/Axum) → vsock → Enclave (Rust)
                              ↑                          ↓
                            IMDSv2                   KMS Decrypt
                                                 (via vsock proxy)
```

**Communication boundaries:**
- API ↔ Parent: HTTPS via Network Load Balancer
- Parent ↔ Enclave: vsock with an 8-byte little-endian length prefix + JSON payload (`parent/src/protocol.rs` ↔ `enclave/src/protocol.rs`). Max message size 10 MB.
- Enclave → KMS: vsock proxy forwarding to HTTPS (the enclave has no direct network access)

The Parent is the only thing that can talk to the Enclave (over vsock). The Enclave must always return error states back to the Parent over vsock — never panic, never silently drop a request. Error messages sent back are sanitized (truncated to 200 chars in `enclave/src/main.rs::sanitize_error_message`) to avoid leaking sensitive data.

**Decrypt data flow:**
1. Lambda retrieves the encrypted vault from DynamoDB
2. Lambda calls the parent over HTTPS
3. Parent extracts IAM credentials from IMDSv2 (cached, auto-refresh 60s before expiry)
4. Parent forwards the request to the enclave over vsock
5. Enclave calls KMS to decrypt the symmetric key, then uses HPKE (RFC 9180: P-384, HKDF-SHA384, AES-256-GCM) to decrypt each attribute (parallelized with `rayon`)
6. CEL expressions optionally transform decrypted data before it's returned

## Cross-Component Sync Points

These pairs MUST be updated together — they implement opposite sides of the same wire format:

| Change | Files |
|--------|-------|
| HPKE encryption/decryption | `api/src/app/encryptors.py` ↔ `enclave/src/hpke.rs` |
| Request/response models | `api/src/app/models.py` ↔ `enclave/src/models.rs` |
| Vsock framing protocol | `parent/src/protocol.rs` ↔ `enclave/src/protocol.rs` |
| CEL functions | `enclave/src/expressions.rs` + `enclave/src/functions.rs` |
| New API routes | `api/src/app/routers/` |

The KMS key policy is the security root of trust: it only allows decryption from enclaves with the expected PCR measurements.

## Build Targets & Runtime

- **Enclave**: `aarch64-unknown-linux-musl`, statically linked, uses `mimalloc` (musl's default allocator is too slow). Release profile in root `Cargo.toml`: `opt-level=z`, `lto=true`, `strip=true`, `panic=abort`, `codegen-units=1`.
- **Parent**: `aarch64-unknown-linux-gnu`, dynamically linked.
- Both crates use Rust edition 2024.

## Code Style

- Line length: 120 characters (Python via Black, Rust via `cargo fmt`)
- Indentation: 4 spaces for Python/Rust, 2 spaces for YAML/JSON
- All source files require an `MIT-0` license header
- Use `zeroize::Zeroizing<T>` (or `#[derive(Zeroize)]`) for any in-memory secret in Rust
- Prefer `thiserror` for library error types, `anyhow` for application errors
- No `unwrap()` / `expect()` / panic-prone macros in enclave or parent production code; `check_no_panic.sh` enforces this and runs in CI

## CI/CD

GitHub Actions, all on `ubuntu-24.04-arm`:
- `rust.yml`: fmt --check, build, clippy `-D warnings`, `check_no_panic.sh`, `cargo test`
- `security_audit.yml`: daily RustSec audit + on every `Cargo.toml`/`Cargo.lock`/`deny.toml` push + manual `workflow_dispatch`
- `docker-bake.yml`: multi-platform Docker builds (no push)
- `deploy-docs.yml`: MkDocs site → GitHub Pages
- `secure_workflows.yml`: workflow security checks

`buildspec.yml` drives the CodeBuild side of the AWS deployment pipeline. Action SHAs are pinned with version comments.

## Dependency Policy

Dependencies are pinned to exact versions (`=X.Y.Z`) for reproducibility. Avoid introducing new dependencies — prefer the Rust standard library where possible. When a new dependency is genuinely needed, pin it exactly and minimize features (most existing deps use `default-features = false`).
